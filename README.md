## 📖 Опис проєкту
Цей проєкт — базова реалізація REST API для керування кулінарними рецептами. Головна мета роботи: побудова правильної архітектури, реалізація бізнес-логіки (Use Cases) та жорстка валідація даних (Інваріанти) з використанням сучасного фреймворку FastAPI.

## 🏗 Структура проєкту
Код розділено на логічні модулі для зручності підтримки та масштабування:

* 📄 `main.py` — **Контролери:** точка входу, маршрутизація HTTP-запитів.
* 📄 `auth.py` — **Безпека:** логіка генерації та перевірки JWT (Bearer Token).
* 📄 `schemas.py` — **DTO та Інваріанти:** Pydantic-моделі для валідації вхідних даних.
* 📄 `test_main.py` — **Тестування:** автоматичні інтеграційні тести.
* 📁 `docs/use-cases.md` — **Документація:** формальний опис прецедентів.

---

## 🛡 Реалізовані Інваріанти (Бізнес-правила)

Система не пропускає невалідні дані (повертає помилку `422 Unprocessable Entity`):

**Сутність "Користувач" (User):**
* Формат `email` суворо перевіряється на відповідність стандартам.
* Довжина пароля має бути **не менше 6 символів**.

**Сутність "Рецепт" (Recipe):**
* Назва страви (`title`) не може бути порожньою.
* Час приготування (`cooking_time`) має бути строго **більшим за 0**.

## 🚀 Як запустити локально

### 1. Встановлення залежностей
Переконайтеся, що у вас встановлено Python, та виконайте команду в терміналі:
```bash
pip install fastapi uvicorn PyJWT pydantic[email] pytest httpx
```

### 2. Запуск сервера
Запустіть локальний сервер за допомогою uvicorn:
```bash
uvicorn main:app --reload
```

Після успішного запуску інтерактивна документація (Swagger UI) з можливістю тестування API буде доступна за адресою:  
👉 **[http://127.0.0.1:8000/docs](http://127.0.0.1:8000/docs)**

У проєкті реалізовано інтеграційні тести (Pytest), які покривають як успішні сценарії (Happy Path), так і перевірку захисту (401 Unauthorized) та інваріантів (422, 409 Conflict).

## 🧠 Питання для самоаналізу

### 1. Архітектура
* **Де живе бізнес-логіка?** У поточній реалізації бізнес-логіка (перевірка інваріантів) зосереджена у **Pydantic-схемах** (`schemas.py`), де описані правила для email, паролів та параметрів рецептів. Прийняття рішень щодо прав доступу (наприклад, перевірка автора при видаленні) відбувається безпосередньо в **контролерах** у файлі `main.py`.
* **Як пов'язані моделі БД і бізнес-логіка?** Оскільки це baseline-версія, ми використовуємо **In-memory сховище** (словники Python). Моделі Pydantic виступають одночасно і як об'єкти для передачі даних (DTO), і як інструмент забезпечення бізнес-правил, безпосередньо взаємодіючи з даними у словниках.
* **Наскільки легко було б замінити базу даних?** Заміна сховища (наприклад, з In-memory на PostgreSQL) потребуватиме змін у багатьох функціях `main.py`, оскільки контролери мають пряму залежність від глобальних змінних `users_db` та `recipes_db`. Це свідчить про високу зв'язність, яку потрібно буде усунути шляхом впровадження паттерну "Репозиторій".
* **Наскільки складно було тестувати?** Тестування було простим, оскільки логіка валідації в `schemas.py` повністю ізольована. Unit-тести бізнес-логіки можна запускати миттєво без підняття будь-якої інфраструктури чи бази даних.

### 2. Масштабування
* **Наскільки легко було б масштабувати сервіс?** Поточна архітектура **не підтримує горизонтальне масштабування**. Якщо запустити кілька екземплярів (інстансів) сервісу, дані між ними не будуть синхронізовані, оскільки кожен матиме власну "базу" в оперативній пам'яті.
* **Що заважає масштабуванню?** Головна перешкода — наявність **стану (state)** у вигляді глобальних змінних та кешу в пам'яті. При зростанні навантаження першими зламаються механізми інкременту ID через стан гонитви (race conditions) та вичерпається RAM.
* **Які архітектурні рішення варто змінити?** Необхідно перейти до **stateless-архітектури**: винести дані у зовнішню БД, прибрати глобальні змінні та впровадити шари абстракції між API та сховищем даних.
